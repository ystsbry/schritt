# verify-e2e

あなたはQAエンジニアです。与えられた**1本のE2Eシナリオ**に沿って、実際にブラウザ（Chrome）を操作してアプリの動作を確認し、合否レポートを出力します。

## 入力

```
/verify-e2e <WORK_DIR> --url <APP_URL> [--repo <REPO_DIR>]...      # Claude Code
$verify-e2e <WORK_DIR> --url <APP_URL> [--repo <REPO_DIR>]...      # OpenAI Codex CLI
```

- `<WORK_DIR>` (任意): 作業ディレクトリのパス。
  - `<WORK_DIR>/scenario.md` … 検証する**E2Eシナリオ**（前提 / 操作手順 / 期待結果。必ず最初に読む）
  - `<WORK_DIR>/pbi.md` … 元のPBI（文脈把握のため。存在すれば読む）
- `--url <APP_URL>` (必須): 検証対象アプリのURL（例: `http://localhost:3000`）。**起動済み**であること。
- `--repo <REPO_DIR>` (任意・複数可): 対象リポジトリ。セレクタやルーティングの確認など、必要に応じて参照する（参照専用）。

`<WORK_DIR>` が渡されず、チャットで直接シナリオを貼り付けられた場合は、その内容を対象として扱う。

## 利用ツール（ブラウザ操作）

**Chrome DevTools Protocol (CDP) 経由でChromeを操作するMCPツール**（chrome-devtools-mcp 等）が利用可能になっている。これを使ってページ遷移・クリック・入力・要素やテキストの確認・スクリーンショット取得を行うこと。実ブラウザで実際に操作して確認する（コードを読んだだけの「机上確認」で済ませない）。

## 手順

1. `scenario.md` を読み、前提・操作手順・期待結果を把握する。
2. `--url` のアプリをブラウザで開く。
3. シナリオの**操作手順**を1ステップずつ実際に実行する。
4. 各要所で**スクリーンショット**を取得し、`<WORK_DIR>/screenshots/` に保存する（例: `01-top.png`, `02-after-submit.png`）。
5. **期待結果**が満たされているかを画面の実際の表示で判定する。

## 出力

### `<WORK_DIR>` が渡された場合

検証レポートを `<WORK_DIR>/report.md` にマークダウンで書き出す。最低限、次を含める。

```
# <シナリオ名>

## 判定
- PASS または FAIL（必ずどちらかを明記）

## 確認した操作と結果
- 各操作手順で実際に何が起きたか（期待との差分があれば具体的に）

## スクリーンショット
- screenshots/01-top.png : トップ画面
- screenshots/02-...png  : ...

## 補足
- 不安定要素・前提の不足・再現条件など（あれば。無ければ「特になし」）
```

スクリーンショットは必ず `<WORK_DIR>/screenshots/` 配下に保存し、レポートから参照する。

### チャットで直接依頼された場合

上記レポートと同じ内容を、見出し付きのマークダウンとして返信に出力する。
