# implement-step

あなたは経験豊富なソフトウェアエンジニアです。与えられた**1つの実装ステップ**を、対象リポジトリに実装し、レポートを出力します。

## 入力

```
/schritt:implement-step <WORK_DIR> [--repo <REPO_DIR>]...   # Claude Code (plugin)
$implement-step <WORK_DIR> [--repo <REPO_DIR>]...           # OpenAI Codex CLI
```

- `<WORK_DIR>` (任意): 作業ディレクトリのパス。
  - `<WORK_DIR>/step.md` … 実装する**このステップの計画**（必ず最初に読む）
  - `<WORK_DIR>/pbi.md` … 元のPBI（文脈把握のため。存在すれば読む）
  - `<WORK_DIR>/notes.md` … 補足メモ（存在すれば読む）
- `--repo <REPO_DIR>` (任意・**複数指定可**): 実装対象のリポジトリ。ここに**コードを実装する**。
  複数渡された場合は、ステップ内容に応じて適切なリポジトリに変更を加える。

`<WORK_DIR>` が渡されず、チャットで直接ステップ内容を貼り付けられた場合は、その内容を対象として扱う。

## 手順

1. `step.md` を読み、このステップで何を実装するかを理解する。`pbi.md` / `notes.md` で文脈を補う。
2. 対象リポジトリの既存コードとテストの書き方（ディレクトリ構成・命名・テストフレームワーク）を調査する。
3. **このステップの範囲のみ**を実装する（他ステップの作業に踏み込まない）。
4. 実装に対応する**単体テスト**を書く。
5. 可能であればテスト/ビルドを実行し、通ることを確認する。

## 出力

### `<WORK_DIR>` が渡された場合

このステップのレポートを `<WORK_DIR>/report.md` にマークダウンで書き出す。最低限、次の見出しを含める。

```
# <ステップ名>

## 実装した内容
- 変更/追加したファイル、追加した関数・型、設計上の判断

## 書いた単体テスト
- 追加したテストファイルとテストケース
- テストを実行できた場合はその結果（pass/fail）

## 補足
- 未対応・TODO・POへ確認したい点など（あれば。無ければ「特になし」）
```

リポジトリへの実装（コード変更）と、`<WORK_DIR>/report.md` の生成の**両方**を必ず行うこと。

### チャットで直接依頼された場合

上記レポートと同じ内容を、見出し付きのマークダウンとして返信に出力する。
